A failed ECS deployment on March 8, 2026 caused repeated task cycling for over 5 hours before a manual rollback restored service.

Summary

On March 8, 2026, a new deployment to the lobbyfix-webapp service on the regiztra-2-prod ECS cluster failed to pass ALB health checks. The new tasks started successfully but died within 18–24 seconds of boot — silently, with no error output. ECS cycled through task replacements approximately every 11 minutes for over 5 hours. Service was restored via a manual rollback to the previous stable image (regiztra-2-webapp:251), which has been running without issue since February 27.

Root Cause

The new image's process exited immediately after startup.

All 18+ tasks from the failed deployment followed an identical pattern:

  1. Container starts (~0s)
  2. App logs Express server listening on port 4000 (~18–20s)
  3. Process exits — no error output, no further logs (~18–24s total lifetime)

The app runs internally on port 4000 but the container maps port 80. A reverse proxy (nginx) bridges the two. The new image almost certainly broke this bridge — either by removing nginx, misconfiguring the entrypoint, or attempting to bind Node.js directly to port 80 (which fails silently under a non-root user with EACCES).

Because the task died before the ALB's first health check at 30s, every probe hit a dead endpoint and returned a timeout.

Contributing Factors

Health Check Configuration Is Unforgiving

The ALB target group is configured with very tight thresholds:

Parameter Value Impact
Interval 30s First check at 30s — task already dead
Timeout 5s Too short for SSR cold start under load
UnhealthyThreshold 2 Only 2 failures needed to mark unhealthy
HealthyThreshold 5 150s of consecutive passes required to become healthy

Even if the process had stayed alive, this configuration would make any slow-starting Node.js app fragile on deployment.

No ECS Container Health Check Configured

There is no healthCheck defined in the task definition. A container-level health check with a startPeriod would:

  • Provide a grace window during startup
  • Give ECS direct signal that the process is alive
  • Reduce false positives from ALB timing

No startPeriod Grace Window

ECS registers the task with the ALB immediately on start. Without a startPeriod, health checks begin before the app is ready.

Findings

  • 18+ task restarts over ~5 hours on March 8
  • Every task from the failed deployment lived exactly 18–24 seconds (matching startup time)
  • Current stable task (:251) has been running continuously since Feb 27
  • The failure was in the new image, not in ECS or ALB configuration

Recommendations

1. Fix the New Image — Port 80 Binding

Confirm the new image still ships nginx (or equivalent) configured to proxy :80 → :4000. If the image now attempts to bind Node directly to port 80, either:

  • Run the container as root (not recommended), or
  • Change the container port mapping to 4000 and update the ALB target group accordingly

2. Tighten the Feedback Loop — Add Startup Logs

The process exits without any error output. Add an uncaughtException / unhandledRejection handler and make startup failures loud:

process.on('uncaughtException', (err) => {
  console.error('FATAL uncaughtException:', err);
  process.exit(1);
});

3. Tune Health Check Thresholds

Make the ALB health check more tolerant of cold starts:

aws elbv2 modify-target-group \
  --target-group-arn arn:aws:elasticloadbalancing:us-east-1:792433572809:targetgroup/regiztra-prod2-webapp/db234ad614ea13c6 \
  --health-check-timeout-seconds 10 \
  --unhealthy-threshold-count 3 \
  --healthy-threshold-count 2

4. Add a Container Health Check with startPeriod

In the task definition, add:

"healthCheck": {
  "command": ["CMD-SHELL", "curl -f http://localhost:4000/ || exit 1"],
  "interval": 10,
  "timeout": 5,
  "retries": 3,
  "startPeriod": 60
}

This gives the app 60 seconds to start before health check failures count against it.

5. Enable ECS Deployment Circuit Breaker

If not already enabled on the service, turn on the circuit breaker with automatic rollback:

aws ecs update-service \
  --cluster regiztra-2-prod \
  --service lobbyfix-webapp \
  --deployment-configuration "deploymentCircuitBreaker={enable=true,rollback=true}"

This would have triggered the rollback automatically instead of requiring manual intervention after 5+ hours.

Security Note

The current task definition (:251) contains AWS_ACCESS_KEY_ID and AWS_SECRET_ACCESS_KEY as plain-text environment variables. These credentials are visible in any describe-task-definition API call, CloudTrail logs, and the AWS Console.

Action required: Rotate those keys and migrate to an IAM task role. This eliminates static credentials entirely and follows AWS least-privilege best practice.
